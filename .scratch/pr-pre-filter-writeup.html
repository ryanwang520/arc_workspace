大目录公司的 opt-in 产品预筛选

19 文件 +4 260 / −29 分支 feat/new_pre_select_approach → main 作者 @haowei 6 个 commit（Phase 0–5）

用户可见效果：给 catalog 在 5k–20k SKU 量级的公司加一个 opt-in 的预筛选环节。开关为 org_profiles.pre_filter_enabled，默认 false——已上线的 org 行为完全不变。开启之后，search 节点仍然加载全量 catalog，但每个 proposal 在调用 select_and_calculate 之前会先按客户在 Q&A 中表达的 material × product_type × height × color 把候选集缩到对应子集，select prompt 的 token 占用和注意力分散都会显著下降。

本 PR 实现 ADR 0002-product-pre-filter 的 Phase 0–5（schema、分类服务、查询时过滤、graph 集成、admin 端点）。Phase 6（observability）和 Phase 7（eval 扩展）有意延后——见滚动发布章节。任何环节出错都 fail-open 退回到全量候选池，pre-filter 不会阻塞 BOM 生成。

为什么做这件事

之前的 embedding-based RAG 已经下线，原因是 top-K 截断会悄悄漏掉冷门 SKU——失败模式是 recall 低，不是检索成本。但全量 catalog 直接喂进 select LLM，对几千到上万个 SKU 的 catalog 来说有四个问题叠加：context 大、慢、贵、效果差（注意力被无关 SKU 稀释）。

这次的设计是故意重新审视预筛选这条路，但用结构上不同的方法：在每个 product 上做闭域集合分类（material × product_type），查询时按 set intersection 过滤，而不是 top-K 相似度。这样 recall 是按类论证的——比黑箱 K 更容易讲清楚。select_product/ 这个 prototype 在 23 个真实订单 / ~21k 产品上验证过 100% recall；这个 PR 借鉴它的形状，重新设计 schema 和流程贴合 arc_agent。

Before / After

BEFORE
  • search_node 加载全量 catalog（最多 ~20k Candidate）
  • select_and_calculate 直接看到所有 SKU
  • 大目录的 select prompt token 占用 / 注意力分散无法控制
  • products 表只有 name, description, unit, embedding
  • 无 per-org 配置，所有 org 走同一条路
AFTER
  • search_node 不变，仍加载全量（即过滤的 fallback 安全垫）
  • 每个 proposal 在 asyncio.gather 内先过 maybe_pre_filter
  • 过滤后的子集才进 select_and_calculate
  • 新增 5 个分类列 + classified_at（feet 单位、TEXT[] material）
  • org_profiles.pre_filter_enabled opt-in；admin 端点切换 + 触发 backfill
  • 任何错误（LLM 异常 / sanity 失败 / 未启用）→ fail-open 退回全量

apps/agent/src/agent/pre_filter.py +400 重点看

整个 PR 的大脑。maybe_pre_filter 是入口；先读 org 的 opt-in flag，把 plan_markdown / proposal_description / user_modification 拼成单段 qa_text，单次 LLM 调用提取 RequiredFilters，再按 ADR 的 4 条 deterministic safety rules 收紧：

  • baseline ∪= {surface, post}（任何项目都需要）
  • 客户提到 gate ⇒ ∪= {gate_hardware}（gate 和 gate 五金永远成对）
  • required_materials ∩ {wood, vinyl} 非空且要 gate ⇒ required_materials ∪= {chain_link}（木 / 乙烯门框是镀锌钢）
  • 提到 barbed / razor 且 required_heights 非空 ⇒ 每个 h 加上 (h−1)（commercial 7+1 标准）

过滤分四个独立轴 AND 起来，每一轴都对产品侧空值兜底（applicability semantics——空 = 通配）。重点看 _filter_by_height 按 product_type 分发的逻辑，以及 _filter_by_color 对 {gate_hardware, access_control, accessory, service, material} 的豁免（黑色合页能放进白色围栏项目）。

_FUNCTIONAL_ALWAYS_PASS  = frozenset({"accessory", "service", "material"})
_FUNCTIONAL_BASELINE_REQ = frozenset({"surface", "post"})
_COLOR_EXEMPT_TYPES      = frozenset({"gate_hardware", "access_control",
                                      "accessory", "service", "material"})
GATE_HEIGHT_TOLERANCE_FT = 1.0

# height 过滤 v1 — recall-first, 仅 surface/gate/post 三档；其它 product_type 一律 pass
# post 这里只做 post_length >= min(required_heights) 的下界；
# 不建模埋深，把深度 vs. 长度的判断交给下游 select LLM。

geometry 优先：当 geometry.fence_run_types[*].height 非空时，会覆盖 LLM 提取的 required_heights（确定性高，零非确定性）；只有 geometry 没有 typed fence runs 时才回落到 LLM 提取。

滚动发布

这是一个完全 opt-in 的功能，默认 pre_filter_enabled=false。合并后所有现有 org 的行为不变，可以放心合入。

启用一个 org 的标准流程

  • STEP 1
    选最小的 opt-in 候选 org
  • STEP 2
    POST /admin/orgs/{org_id}/pre_filter/enable——会 fire-and-forget 后台分类（参考成本：1k 产品约 $0.50 / ~3 分钟，17k 产品约 $3 / ~6 分钟）
  • STEP 3
    GET status 直到 classified ≈ total（≥95%）；手工抽 10 行验证 (name, material, product_type)
  • STEP 4
    真实 query 流量自动开始走预筛选；监控 BOM missed_products 与启用前两周对照

风险隔离

  • fail-open 是承重设计——LLM 异常 / sanity 失败 / DB 故障 / 任何意外 ⇒ 退回到全量候选池（与 opt-out 同路径）。pre-filter 不会阻塞 BOM 生成。
  • 新同步未分类的产品以 wildcard 参与，precision 短期下降而非 recall 下降。
  • patch_bom 是已知盲区——它直接编辑既有 BOM，不需要候选池。ADR 里有记录。

有意延后的事

DEFERRED

Phase 6（结构化日志）和 Phase 7（eval 扩展）暂未实现。意思是：第一个 opt-in org 上线时，会暂时缺少 pre_filter.applied 的结构化指标和 pre_filter_recall 的归因。开任何生产 org 之前，建议先把 Phase 6 加上（10 分钟改动），这样 fallback 率有 alert。

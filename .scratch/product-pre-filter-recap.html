THREAD RECAP · ARC_AGENT

Opt-in product pre-filter for large catalogs

Claude Code session 2026-05-09 multi-day, ~3 phases (grilling → planning → implementation) recapped for arc_agent reviewers

We borrowed the idea of the select_product/ prototype — pre-classify every SKU, filter at query time by intersecting customer requirements — and re-designed the schema and flow to fit arc_agent's constraints. The prior embedding-based RAG was retired because top-K truncation kept losing cold-tail SKUs; we deliberately revisited pre-filtering with closed-domain set classification so recall is reasoned about per-class, not per-K.

The decision tree settled on two axes (Material set × Product type enum), all heights in feet, color participating in filtering, single LLM extraction per proposal, deterministic safety rules, and fail-open on every error. Six commits landed on feat/new_pre_select_approach; Phase 6 (logging) and Phase 7 (eval extension) were intentionally deferred. Independent review returned 0 blockers, 7 fix-worthy concerns, 5 nits.

Decisions & tradeoffs

DECISION 1

How many axes? — Material × Product type, not 9-tag mixed-axis

  • CHOSEN Two axes: Material (set, 4 values) + Product type (enum, 9 values) Material captures the fence system; Product type captures the role. Each axis is closed and reasoned about independently.
  • REJECTED Reuse select_product/'s 9-tag scheme as-is Conflates material (wood, vinyl) with functional role (access_control) on a single axis. Known LLM tag-drift on mixed-axis values.
  • REJECTED Three axes: material + tags + spec attributes Once tags need to be multi-valued (so hinges/latches survive cross-system), tags ≈ a second material set. The third axis collapses into the first two.

Because: two axes give 100% structural coverage of the catalog with zero drift between roles and materials. Cross-axis-restricted SKUs (e.g. a wood-only decorative hinge) are <2% and fall back acceptably to the dominant role.

Tradeoff: a single product can't carry conflicting roles — a "gate operator + barbed top arm combo kit" gets the dominant role assigned, not both.

DECISION 2

Material domain: 4 closed values, no universal token

  • CHOSEN {wood, chain_link, vinyl, ornamental}; empty set = wildcard chain_link consolidates galv + colored mesh (color goes on spec.color). ornamental subsumes aluminum / iron.
  • REJECTED Add universal as a literal value "Wildcard" is a semantic property, not a value. A literal universal token forces the prompt to pick between "all four" vs. "universal" with no clear rule.
  • REJECTED Split chain_link_galv from chain_link_color Coating is a spec dimension, not a material system. Doubling the axis to capture it leaks color into pre-filter.

Because: "applicability" is the property we want to filter on, and applicability fits cleanly as set-on-product-side: a hinge with {wood, chain_link, vinyl, ornamental} means "applies to all four", a project commodity with {} means "applies regardless".

Tradeoff: the value set is fence-industry-closed. Multi-vertical orgs (deck + fence) need an extension to CONTEXT.md first.

DECISION 3

Product type: single-valued, 9 enum values

  • CHOSEN Single-valued: surface | gate | post | gate_hardware | access_control | barbed_wire | accessory | service | material Three groups: form-factor (drives height filter), Q&A-triggered functional roles, and always-pass overhead.
  • REJECTED Multi-valued tags alongside Material Cross-axis SKUs (gate_hardware + access_control) are rare in the fence vertical. Designing for <2% added complexity for negligible recall gain.
  • REJECTED Drop barbed_wire — fold into accessory Barbed wire has a strong Q&A trigger ("razor", "barbed top arm") and pairs with a deterministic height-1ft safety rule. Worth its own enum value.

Because: the LLM is far more reliable on single-value enum classification than on a multi-label structured output, and the failure modes we want to catch (gate without gate_hardware, post too short) all fall out of single-valued routing.

DECISION 4

Storage units: feet everywhere, even where the catalog says inches

  • CHOSEN NUMERIC(4,1) in feet for fence_height and post_length Geometry.fence_run_types[*].height is already feet (per prompts/geometry.py). Storing classification fields in feet means zero conversion at any read site.
  • REJECTED Inches as storage unit (matching catalog naming convention) Fence-industry catalogs name SKUs in inches ("96\" post"). Storing in inches mirrors that, but forces a ×12 conversion every time geometry-derived heights flow into the filter.

Because: a unit-conversion landmine on every read site is exactly the kind of bug that surfaces in production after weeks of working "fine" — better to align the storage unit to the source of truth (geometry) and make classification do the inches→feet conversion at write time only.

Tradeoff: the classification prompt has to be explicit about reporting feet (decimals OK: 6.0, 6.5, 8.0), and the reviewer has to scan it for unit drift.

DECISION 5

One LLM call per proposal — not 3-call union for recall stability

  • CHOSEN Single low-temperature LLM extraction per proposal + deterministic safety rules Per consultation run with N proposals: N LLM calls. Safety rules union baselines ({surface, post}, gate→gate_hardware, etc.) compensate for stability the union was buying.
  • REJECTED 3-call union extraction (per select_product's recall mechanism) Per-proposal × 3 = up to 9 calls per multi-proposal run. Cost scales with proposal count and the union mostly recovers what safety rules can.

Because: deterministic post-extraction rules absorb the recall-stability that union was buying. The remaining LLM error mode (extracting empty required_materials) is caught by the sanity check + fail-open.

DECISION 6

Failed-open everywhere — pre-filter never blocks BOM generation

  • CHOSEN Any error → return the full pool with a fallback_reason LLM exception, sanity violation (required_materials==[] in a fence-context run), opt-out, or any other exception: log + return all candidates. Same path as pre_filter_enabled=false.
  • REJECTED Hard error on extraction failure Pre-filter is a precision-improving optimization, not a correctness gate. Letting it block BOM generation makes it strictly worse than not having it.

Because: recall is the load-bearing property — the previous embedding-based RAG was retired specifically because it failed recall silently. We accept that pre-filter occasionally degrades to "no filtering" rather than ever drops a relevant product.

DECISION 7

Post height filter: conservative post_length ≥ min(required_heights), no buried-depth modeling

  • CHOSEN post_length ≥ min(required_heights), NULL → keep Only short posts get dropped. Depth-vs-length nuance is delegated to the downstream select LLM.
  • REJECTED 1/3 industry rule (post_length ≥ fence_height + ⅓·post_length) Real catalogs already violate it — the standard 6ft fence + 8' post pairing has ~25% buried, not 33%. Strict enforcement drops legitimate SKUs.
  • REJECTED install_method LLM extraction (dig / drive / surface mount) Customers rarely state install method explicitly. Adding an LLM signal that misses 80% of the time wasn't worth the prompt complexity.

Because: burial depth varies by company / install method / region — there's no formula that's right across all orgs. Recall-first is consistent with the rest of the design.

Tradeoff: a 6ft fence will keep an 18ft post in the pool. The select LLM has to do the right pick.

DECISION 8

Color participates in filtering, with hardware/accessory exempt

  • CHOSEN Substring match either direction; {gate_hardware, access_control, accessory, service, material} exempt "Satin Black" ⊃ "Black" survives. NULL on either side = wildcard. A black hinge fits a white fence project.
  • REJECTED Exact-string match on color Catalog colors have decorator words ("Bronze Fine Texture", "Satin Black") that the customer never types verbatim. Exact match drops legitimate SKUs.
  • REJECTED Skip color filter entirely (defer to select LLM) Color is a strong signal when stated. Letting the candidate set carry colors that don't match wastes select-prompt tokens for no benefit.

Because: when the customer states a color, it's typically a hard constraint on visible surfaces but cosmetic on hidden hardware. The exempt set lines up with "you can't see it from across the yard".

DECISION 9

Where filtering runs: in the executor nodes, not in the search node

  • CHOSEN maybe_pre_filter inside run_consultation_proposals (per-proposal) and run_regenerate_select search_node stays thin: fetch_all_candidates only. Filtering happens just before select_and_calculate.
  • REJECTED Filter inside search_node Per-proposal context (each proposal's description) isn't available in search. Filtering once for all proposals would either over-filter or under-filter for individual proposals.
  • REJECTED SQL-side filter (PostgreSQL && overlap) Deferred to v2. 20k Candidates fetched into Python adds hundreds of ms but keeps search and state-shape unchanged. Optimize if measured latency becomes a problem.

Because: per-proposal filtering matches the per-proposal Q&A signal each one carries. patch_bom and run_regenerate_calculate intentionally skip pre-filter — they don't select from scratch.

Dead ends

TRIED

Treating the existing select_product/ prototype's 9-tag scheme as the reference for arc_agent's pre-filter.

Why we stopped: the prototype's 9 tags mix material with role on a single axis. LLM tag drift on mixed-axis values is documented as a known failure (select_product/EXPERIMENT.md Phase 7.5). The reusable insight was the shape (closed-domain set classification + intersection at query time), not the schema.

TRIED

Adding a separate multi-valued tags field next to material for cross-system items like hinges, latches, gate hardware.

Why we stopped: once the tags set is multi-valued for cross-system support, it duplicates the structure of material. Better to put functional roles inside product_type as accessory subclasses (gate_hardware, access_control, barbed_wire).

TRIED

Hooking classification onto upsert_product_library (the catalog sync HTTP path) so newly-synced products are immediately filterable.

Why we stopped: would block the sync HTTP path on slow LLM calls. We took the embeddings-pattern path instead: async backfill + a periodic scan for classified_at IS NULL rows. New products fall through as wildcards until classified.

TRIED

Encoding a buried-depth rule (1/3 industry standard, fixed +24" assumption, install_method LLM extraction).

Why we stopped: every concrete formula either over-filters real catalogs (strict 1/3 drops the standard 6ft + 8' post pairing) or requires a customer-stated install_method that's almost never explicit. The conservative post_length ≥ min(heights) got us recall safety; depth nuance moved to the select LLM.

TRIED

Storing classification heights in inches to match catalog naming convention.

Why we stopped: Geometry.fence_run_types[*].height is in feet. An inch-storage choice would put a ×12 conversion landmine on every read site (filter, sanity check, geometry override). We standardized on feet across schema, candidate, prompt, and filter.

TRIED

Embedding-based RAG (LLM-generated queries + pgvector top-K). This was retired before this thread but is the failure mode we were specifically avoiding.

Why we stopped: top-K truncation silently dropped relevant cold-tail SKUs. The failure mode was recall, not retrieval cost. Set-intersection on closed-domain classes makes recall reasoning per-class instead of trusting a black-box K.

Open questions / known concerns

Independent code review returned 0 blockers. Items below are worth resolving but don't block merge.

  • FIX Per-proposal _is_pre_filter_enabled SELECT inside asyncio.gather — N proposals = N concurrent SELECTs for the same row. Read once in run_consultation_proposals_node and pass the bool through. → small cleanup commit
  • FIX Doc comment in _extract_requirements claims sanity check short-circuits the LLM call for empty Q&A — it doesn't (sanity check runs after extract). Fix the comment or add the actual short-circuit.
  • FIX ARRAY(String) in the model vs ARRAY(Text()) in the migration — functionally equivalent in PG, but autogenerate will flag spurious diffs on the next alembic revision.
  • FIX composite appears in _METAL_GATE_FRAME_MATERIALS (per ADR rule 3) but the canonical 4-material domain has no composite. Either drop from ADR or extend material domain.
  • FIX _FUNCTIONAL_CLIENT_GATED is defined but never read — dead code.
  • FIX Bare except Exception in test_product_classification.py:81 silently maps SQLAlchemy internals to {} — could let bugs pass tests as "no values asserted". Narrow or remove.
  • DEFER Hard-coded get_llm("gemini-3-flash-preview", "minimal") in pre_filter.py — not threaded through BomStageName. Add a pre_filter stage or document the carve-out.
  • DEFER Phase 5 admin endpoints reuse verify_internal_service_token (the only auth gate that exists). Decide whether opt-in toggles need a stricter human-admin path before production rollout.
  • DEFER Phase 6 (structured logging on pre_filter.applied) and Phase 7 (eval extension with pre_filter_recall) skipped. Means flying blind on observability and attribution at first opt-in. Consider adding before any production org goes live.
